# 架构说明

GameBuddy Agent 采用轻量级多代理流水线，目标是：

- 本地开发简单
- MVP 可完整运行
- 架构清晰，便于开源协作
- 后续可逐步替换为更强的感知或检索模块

## MVP 为什么选“类宝可梦回合制对战”

首个 Demo 选择类宝可梦回合制对战，原因很直接：

- 战局天然适合结构化 JSON 表达
- 不依赖脆弱的私有 API
- 适合做“下一步行动建议”和“新手解释”
- 很适合后续接入截图感知层

这个选择优先保证了：

- 可演示性
- 可维护性
- GitHub 展示质量
- 架构的后续扩展空间

## 核心流程

1. 前端提交结构化战局 JSON，或上传截图并附带用户问题
2. Perception Agent 将输入统一整理成 `BattleState`
3. Knowledge Agent 从本地知识库加载术语、原型、基础教学信息
4. Strategy Agent 基于局势和知识上下文生成战术建议
5. Review Agent 生成面向复盘的总结内容
6. Orchestrator 将多个代理结果组合成统一响应结构返回前端

## 模块职责

### Perception Agent

负责输入规范化。

当前 MVP 中：

- JSON 输入是完整支持的
- 截图输入是诚实占位实现

后续可以替换成：

- OCR
- HUD 信息抽取
- 简单目标识别
- 多模态模型推理

### Knowledge Agent

负责本地知识组织与查询。

当前使用：

- 本地 JSON 知识库
- 原型说明
- 新手提示
- 战术术语解释

后续可以升级为：

- Markdown 知识包
- SQLite 索引
- Embeddings / RAG
- 多游戏知识包切换

### Strategy Agent

负责根据局势做“下一步建议”。

当前实现方式：

- 以规则和启发式为主
- 语言面向用户输出
- 主动暴露不确定性

它的目标不是假装全知，而是在信息不完整时给出更稳妥、更可解释的建议。

### Review Agent

负责把当前局势转成“教练式复盘”。

输出重点包括：

- 当前处境
- 可能的失误
- 推荐动作
- 长期训练建议

### Orchestrator

负责编排整个调用链，保证：

- 各代理解耦
- API 输出稳定
- 前端无需知道内部实现细节

这让后续替换单个代理模块时，不需要重写整个系统。

## 设计原则

- 诚实的 Mock 优于虚假的 AI 魔法
- 清晰边界优于过度封装
- 本地知识优先，远程依赖后置
- 输出服务于教学解释，不服务于自动化操作
- 先做小而完整的 MVP，再逐步增强能力

## 当前已知边界

- 截图分析不是完整视觉识别
- 不接入真实游戏客户端
- 不读取进程内存
- 不注入输入
- 不实现自动操作逻辑

这些限制不是缺陷，而是项目定位的一部分：  
GameBuddy Agent 是一个策略分析与复盘工具，而不是游戏自动化系统。
